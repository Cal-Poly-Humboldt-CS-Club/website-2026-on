## Getting jobs: 
[leetcode.com](https://leetcode.com) - Ben talked about what leetcode problems look like. They are the standard used for FAANG interviews, so experience with them is recommended. Leetcode problems are very abstract, and rarely come up in day-to-day software engineering. However, they are used because it sets a minimum bar of knowledge, and it is very standardizable (doesn't vary based on interviewer)

[coderpad.io](https://coderpad.io) - a website specifically designed for virtual leetcode problem style interviews. The idea is you and interviewers both see the code and you talk them through the problem. The interviewer is there to help you out, and see how you are to work with.

Ben recommends:
- learning test driven design (which is writing the code in a way that is easy to test I believe?).
- doing things to stand out. You don't want to solve it the same way as everyone else.

## Preparing for leetcode interviews:
There are only 18 real problem types, the rest are variations on those. Developing templates in your head and fitting it to a problem is a good idea. Write it the same.

[Grind75](https://www.techinterviewhandbook.org/grind75/) is a website that helps you to practice Leetcode problems. Just a few a day is all you need. Ben says it takes a few problems a day for about 2 months to get good at Leetcode, then you get the job, and can forget about it.
There is an advantage of Leetcode for junior programmers: you can get really good at Leetcode as fast as senior programmers, as leetcode skills don't carry over into real work applications.

When you are going for an entry level position, easy level leetcode questions are what you should really focus on, as the chance of getting a medium question is unlikely. Speed is what is very important.

## Applying for jobs:
> "Job search is a full time job, it just doesn't pay you."

Maintain a strict schedule: start at a set time and end at a set time so it doesn't consume your real life and burn you out.
Current steps to get a job (job search is hard so don't get demoralized. Each time you go through the interview process you learn):
1. Apply (you will likely not get a reply to these for most jobs. For those where you do get replies, most will be denials)
2. Recruiter Screening (a nontechnical person is the recruiter, it is easy to get past this step)
3. Initial Screening (first technical screening, FAANG companies often ask for no ai, as they want to see how you think. An interview is a date. Ask them questions. The goal is to learn about the company)
4. Coding Interviews
5. Interview loop (3-4 interviews on the same day)
    - Behavior interviews (google for Amazon behavior interview guidance. Prep stories beforehand so you have a lot to go off of. Talk about past projects (Ben made a slideshow to talk about it). Talk about the work, and talk about the impact it had on the company: it shows you have been trusted with responsibility and succeeded)
    - System design (inconsistent term, only sometimes used for junior programmers. You don't have to know it as a junior programmer, but it makes you stand out. To learn: google "hello interview system design: hellointerview.com". The idea is you design a large, interconnected system. This is basically Software Engineering 101).

A referral/connection is the most important thing you could have. It gets you around this process.

Mock interviews are good idea. Darrell Ross (@eukota on discord) suggested that he would work with people on mock interviews, so reach out to him. Darrell also said he would be willing to give referrals for the company he works for, Axiom. Using AI tools which apply to many jobs for you is effective, Ben recommends [applypass.com](https://applypass.com) for that

LinkedIn uses a priority queue, depending on how likely you are to respond, so respond to all messages.

## Life on the job:

Everybody is using AI to write code these days. You should learn how to, however it is fairly easy. It takes about 2 weeks to learn.
Ben uses it heavily to learn a codebase. It is good at learning the scope of a technical implementation and understanding an entire project. It can explain why something is done the way it is.

It tends to write kinda goofy code a lot of the time, and you should review all of the code it writes. It is also good at teaching you new languages: Ben learned it to help him with Rust, but he wouldn't have found it as easy to learn if he wasn't familiar with C family libraries. Ben uses AI to help him learn. One trick he uses is to ask the AI for a plan, to help him learn things in a correct and productive order. He also uses it to quiz him, but he makes sure he gets stuck first before he asks it to explain the solution.
Google Linus Torvalds AI interview, it's worth a watch. The cliff notes are: AI is good at helping you, but you still need the experience to ask it good questions.

- A lot of things are cloud based, so do projects that deal with the cloud when you can. Look into AWS certificates, it shows you have experience. Certificates are not as important as experience, though
- Pull requests are the daily lifeblood of contemporary software development. Learn git in school while you can. Having experience working on a project while other people are working on it is very very good.
